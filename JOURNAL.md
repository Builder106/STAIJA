## 2026-05-27 — Staging silently broken since `fe760e6` (manualChunks TDZ); lazy-init refactor attempted, reverted #incident #decision

Tried to lazy-load the Firestore/Storage/Functions SDKs out of the eager bundle ([239b965 perf(firebase): lazy-load firestore/storage/functions SDKs]). Build green, vue-tsc green, 185/185 vitest green — pushed to staging. The earlier `perf: split avatar/parts` commit ([fe760e6]) had added a `manualChunks` config that emitted a chunk graph with a circular static dep all along — staging had been silently broken since `fe760e6` shipped, but nobody opened staging in a browser between then and now, so the regression went unnoticed for ~1 day. Prod (`main` at `5244687`) was spared only because the perf series had never been promoted there. ## 2026-05-26 — Avatar `parts.ts` (20 MB of inline SVG) was leaking into the main bundle, causing 8 s mobile FCP #incident #decision

Vercel Speed Insights showed Real Experience Score 30/100 on mobile (Poor), driven by 8 s FCP / 8.29 s LCP on average West African mobile networks. Root cause traced via `rollup-plugin-visualizer` + sourcemap inspection: `src/services/avatar/parts.ts` is a 20 MB generated file containing the SVG path data for 10 layered-avatar portraits (the paused animated-avatar feature). It was eagerly imported by `src/services/avatar/index.ts` via `PORTRAIT_SLOT_COUNT = PORTRAITS.length`, which forced parts.ts to load at module init even though no eagerly-loaded code actually rendered SVGs. SiteHeader.vue's `import { resolveAvatarSrc }` from that module dragged the full library into the main chunk — every page load on every device. Fixed by splitting `services/avatar/` into `index.ts` (URL-only helpers, hardcoded `PORTRAIT_SLOT_COUNT = 10`, no `./parts` import) and a new `svg.ts` (SVG-rendering helpers, imports parts.ts). All current consumers use URL-only helpers, so the parts.ts data now has zero importers in the eager graph — it disappears from the production bundle entirely. Main bundle dropped from **20,205 KB raw / 2,934 KB gzipped → 384 KB raw / 63 KB gzipped** (largest chunk). Expected mobile FCP: 8 s → ~1.5–2.5 s. Also added `manualChunks` config in [vite.config.ts](vite.config.ts) splitting Firebase, Tiptap, Contentful, DiceBear, Iconify, Vue framework into independent vendor chunks for cleaner caching, plus a gated `rollup-plugin-visualizer` plugin (`ANALYZE=1 npm run build` opens `dist/stats.html` with a treemap) so future bundle bloat is detectable.

## 2026-05-26 — Cloud Scheduler doesn't support `africa-south1` for either Firebase project; scheduled functions overridden to `europe-west1` #incident #decision

Surfaced during the prod region migration. Both `staija` and `staija-staging` projects refuse `africa-south1` for Cloud Scheduler resources (`gcloud scheduler locations list --project=staija` returns Asia, Australia, Europe, ME, NA, SA, US — no Africa). Likely a legacy App Engine region lock predating Cloud Scheduler's decoupling from App Engine. The 4 `onSchedule` functions (`reapApplicationDraftTombstones`, `reapApplicationStagedFiles`, `reOfferDeferredOnCohortStart`, `sendReferenceReminders`) deployed as Cloud Function code in `africa-south1` but their cron jobs **never registered** — they were silently dead on both staging and prod after the original migration. Worse: Firebase reported "Deploy complete!" on the second pass while leaving the scheduler-side resources missing. Fix: override `region: 'me-central1'` per scheduled function (Doha is geographically closest of the available Scheduler regions to West Africa, ~75–85ms to Lagos vs ~150ms from us-central1; latency cost is negligible anyway because these run daily, not on user-facing paths). Each cron now fires from me-central1 → invokes the function in me-central1 → function calls Firestore in africa-south1 cross-region. Picked Middle East over europe-west1 for the explicit geographic-proximity narrative — when the project's user base is West African, "we keep the entire user-facing path in Africa, with the unavoidable cron carve-out hosted in the Middle East" reads cleaner than "the cron lives in Belgium." Verification step: `gcloud scheduler jobs list --project=<id> --location=me-central1` must show 4 jobs after deploy.

## 2026-05-26 — Design rule: which Cloud Functions can be `onRequest` (public HTTP) vs `onCall` #decision

A function gets `onRequest` (publicly callable HTTP endpoint) only when **all four** of the following hold: (a) the caller is genuinely anonymous — no Firebase Auth session exists at call time; (b) the response is an explicit allowlist of fields, never a full doc dump, so future schema additions stay private by default; (c) any user-influenced behavior is gated on consent flags (`directoryHidden`, opt-in toggles) or one-time tokens issued by a trusted server path; (d) input is validated against a strict shape before any database read. Today the only three endpoints that meet all four are `getPublicMentors` (mentor showcase grid), `resolveReferrerName` (referral hero personalization), and `submitReferenceLetter` (referee uploads via tokenized link). All three sit behind App Check + per-endpoint validation + output-field allowlists. Anything that doesn't meet all four conditions must be `onCall` with explicit role checks — examples already in the codebase: `submitAssignment`, `gradeSubmission`, `enrollStudent`, `deleteAccount`. Adding a fourth `onRequest` endpoint should require a JOURNAL entry justifying why each of the four conditions is satisfied.

Commit `023c584`. Two fire-and-forget paths could lose applicant data without anyone noticing. ## 2026-05-03 — App Check debug token was shipping to prod; analytics had been silently off #incident

Commit `fb0b511`. Prod was 403'ing every Firebase request because the deployed bundle set `self.FIREBASE_APPCHECK_DEBUG_TOKEN = true` — `import.meta.env.DEV` evaluated truthy at build time on Vercel (likely a misconfigured `NODE_ENV`), so the SDK called `exchangeDebugToken` instead of the reCAPTCHA Enterprise exchange and App Check rightly rejected the bogus token. Re-gated the dev path on `window.location.hostname` so it's physically impossible to ship to `staija.org` no matter how the host sets the build mode.
